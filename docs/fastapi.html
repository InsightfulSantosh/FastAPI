What is FastAPI?

FastAPI is a modern Python web framework for building APIs. It is built on Starlette (for HTTP) and Pydantic (for data validation), and uses Python type hints throughout to give you editor autocomplete, runtime validation, and auto-generated docs with zero extra work.

Theory deep dive

An API is a contract between a client and a server. The client sends an HTTP request, the server validates it, runs business logic, and returns a structured HTTP response. FastAPI makes this contract explicit by using Python function signatures and Pydantic models as the source of truth.

This same pattern applies to any domain: users, products, orders, appointments, health profiles, payments, files, or employees. A good FastAPI application makes that contract clear through routes, schemas, status codes, and documentation.

FastAPI compared with other API frameworks

FastAPI is not the only way to build APIs in Python. The best framework depends on the project size, team experience, performance needs, and how much structure you want.

FrameworkBest forStrengthsTradeoffs
FastAPIModern APIs, ML services, microservices, typed backendsFast, async-ready, automatic docs, Pydantic validation, dependency injectionLess built-in admin/auth than Django; you choose more architecture pieces yourself
FlaskSmall apps, prototypes, simple APIsMinimal, flexible, huge ecosystem, easy to learnNo built-in validation/docs/async structure; larger apps need more manual design
Django REST FrameworkLarge database-backed apps, admin-heavy productsDjango ORM, admin panel, auth, permissions, serializers, mature ecosystemHeavier; async story and type-driven docs are not as natural as FastAPI
StarletteLow-level ASGI services and custom frameworksVery lightweight, async-first, powerful primitivesLess high-level convenience; FastAPI already builds on Starlette

Rule of thumb: choose FastAPI when you want Python, strong validation, automatic API docs, async support, and clean API contracts without the weight of a full Django application.

Routing & Request Types

FastAPI maps Python functions to HTTP methods using decorators. Each route describes one operation the API can perform, such as reading a profile, creating an order, uploading a file, or deleting a record.

Theory deep dive

Routing is the process of matching an incoming request to the correct Python function. FastAPI looks at the HTTP method, the URL path, path parameters, query parameters, and request body to decide what function to call and what data to pass into it.

Good route design should make intent obvious. A route like GET /users/101 clearly reads one user, while DELETE /orders/501 clearly removes one order. This clarity matters because frontend developers, API users, tests, and documentation all depend on stable route contracts.

HTTP methods

GET

Read / retrieve data. Safe and idempotent. Use for listings and detail views.

POST

Create a resource. Not idempotent — calling twice creates two records.

PUT

Full replace of a resource. Idempotent. Send the complete representation.

PATCH

Partial update. Only the fields you send are changed.

DELETE

Remove a resource. Idempotent.

Path parameters

Declare a path parameter in the URL string and as a function argument with the same name. FastAPI validates the type automatically.

@app.get("/users/{user_id}", response_model=UserResponse)
def get_user(user_id: int):
    user = find_user(user_id)
    if not user:
        raise HTTPException(status_code=404, detail="User not found")
    return user

# GET /users/101  -> returns user 101
# GET /users/abc  -> 422, because user_id must be an integer

Pydantic & Data Validation

Pydantic models define the shape and rules for API data. They can validate simple fields, nested objects, enums, lists, dates, computed values, and cross-field rules before your endpoint logic runs.

Theory deep dive

Validation protects your application boundary. Without validation, invalid data can enter your business logic, database, logs, and downstream systems. Pydantic checks data before your endpoint logic runs, so your route functions can assume the input already matches the declared schema.

Serialization is the opposite direction: it controls how Python objects become JSON responses. response_model is important because it prevents accidental fields from leaking and keeps response structure predictable.

Computed field example: BMI from height and weight

Pydantic can do more than check fields. It can calculate derived values. For example, if a client sends height and weight, the API can return BMI without asking the client to calculate it.

from pydantic import BaseModel, Field, computed_field, model_validator

class HealthProfile(BaseModel):
    age: int = Field(..., ge=1, le=120)
    height_cm: float = Field(..., gt=30, le=250)
    weight_kg: float = Field(..., gt=1, le=400)

    @computed_field
    @property
    def bmi(self) -> float:
        height_m = self.height_cm / 100
        return round(self.weight_kg / (height_m ** 2), 2)

    @computed_field
    @property
    def bmi_category(self) -> str:
        if self.bmi < 18.5:
            return "underweight"
        if self.bmi < 25:
            return "normal"
        if self.bmi < 30:
            return "overweight"
        return "obese"

    @model_validator(mode="after")
    def check_realistic_child_weight(self):
        if self.age < 10 and self.weight_kg > 120:
            raise ValueError("Weight looks unrealistic for this age")
        return self

@app.post("/health/bmi", response_model=HealthProfile)
def calculate_bmi(profile: HealthProfile):
    return profile
# Request
{
  "age": 30,
  "height_cm": 175,
  "weight_kg": 70
}

# Response includes computed fields
{
  "age": 30,
  "height_cm": 175,
  "weight_kg": 70,
  "bmi": 22.86,
  "bmi_category": "normal"
}

Validators

from pydantic import field_validator, model_validator

class Product(BaseModel):
    name: str
    price: float
    discount: float = 0.0

    @field_validator("name")
    def name_must_not_be_blank(cls, v):
        if not v.strip():
            raise ValueError("Name cannot be blank")
        return v.title()

    @model_validator(mode="after")
    def discount_less_than_price(self):
        if self.discount >= self.price:
            raise ValueError("Discount must be less than price")
        return self

Dependency Injection

FastAPI's Depends() system lets you declare reusable logic such as database sessions, authentication, configuration, permissions, rate limits, and shared service objects.

Theory deep dive

Dependency injection separates setup logic from business logic. Instead of every route manually opening a database session, checking authentication, or reading configuration, the route declares what it needs and FastAPI provides it.

This keeps endpoints focused. For example, a delete endpoint should focus on deleting a resource, not on parsing tokens, opening database connections, checking roles, or reading settings. Dependencies make those cross-cutting concerns reusable and testable.

Database dependency pattern

from fastapi import Depends
from sqlalchemy.orm import Session

def get_db() -> Session:
    db = SessionLocal()
    try:
        yield db
    finally:
        db.close()

@app.get("/records", response_model=list[RecordResponse])
def list_records(db: Session = Depends(get_db)):
    return db.query(Record).all()

Router-level dependencies

Use yield in a dependency to get setup/teardown behaviour (like a context manager). Code before yield runs before the request; code after runs after the response is sent.

Caching — use_cache

# is only executed ONCE (result is cached per request).
# To disable this:
Depends(get_db, use_cache=False)

Error Handling

FastAPI provides several layers of error handling. Pydantic handles invalid request bodies automatically, while your application code should handle business-rule errors and missing resources intentionally.

Theory deep dive

Error handling is part of API design, not an afterthought. A good API tells the client whether the request failed because of invalid input, missing data, missing authentication, insufficient permission, or a server-side bug.

Use 4xx errors when the client can fix the request. Use 5xx errors when the server failed unexpectedly. Common examples are 400 for a business-rule violation, 404 for a missing resource, and 422 for validation failure.

Business-rule error: 400 Bad Request

from fastapi import HTTPException

@app.post("/products", response_model=ProductResponse)

Custom exception class + handler

from fastapi import Request
from fastapi.responses import JSONResponse

class ResourceNotFoundError(Exception):
    def __init__(self, resource: str, resource_id: int):
        self.resource = resource
        self.resource_id = resource_id

@app.exception_handler(ResourceNotFoundError)
async def handle_not_found(request: Request, exc: ResourceNotFoundError):
    return JSONResponse(
        status_code=404,
        content={"detail": f"{exc.resource} {exc.resource_id} not found"},
    )

Override default validation error handler

from fastapi.exceptions import RequestValidationError

@app.exception_handler(RequestValidationError)
async def validation_exception_handler(request: Request, exc: RequestValidationError):
    return JSONResponse(
        status_code=422,
        content={"message": "Request validation failed", "errors": exc.errors()},
    )

HTTP Status Codes Reference

Status codes are part of your API contract. They tell the client whether a request succeeded, failed because of client input, or failed because of a server problem.

Theory deep dive

Use status codes consistently. A frontend, mobile app, or another backend service often makes decisions based on status codes before reading the response body. Good status code choices make APIs easier to debug and integrate.

FastAPI automatically returns 422 for validation errors, but you should choose explicit codes for business behavior such as duplicate resources, missing records, unauthenticated requests, and forbidden actions.

Advanced Topics

Theory deep dive

Advanced FastAPI features solve problems that appear when the API grows: frontend access, request tracing, file uploads, real-time updates, long-running side effects, database persistence, and deployment. You should add these features only when the project actually needs them.

A sensible progression for most APIs is: add database persistence, add authentication for write operations, add tests, add CORS for a frontend, then add background tasks, file uploads, or WebSockets only if the product needs them.

Middleware

Middleware wraps every request and response. It is useful for request logging, timing, trace IDs, rate limiting, compression, or adding security headers.

from starlette.middleware.base import BaseHTTPMiddleware
import time

class TimingMiddleware(BaseHTTPMiddleware):
    async def dispatch(self, request, call_next):
        start = time.time()
        response = await call_next(request)
        response.headers["X-Process-Time"] = str(time.time() - start)
        return response

app.add_middleware(TimingMiddleware)

CORS for frontend apps

from fastapi.middleware.cors import CORSMiddleware

app.add_middleware(
    CORSMiddleware,
    allow_origins=["http://localhost:3000", "https://app.example.com"],
    allow_credentials=True,
    allow_methods=["GET", "POST", "PUT", "DELETE"],
    allow_headers=["*"],
)

Background tasks

Use background tasks for work that should happen after the API response, such as sending emails, writing audit logs, calling webhooks, or refreshing cached data.

from fastapi import BackgroundTasks

def send_confirmation_email(email: str):
    print(f"Sending confirmation email to {email}")

@app.post("/signup")
def signup(payload: UserCreate, bg: BackgroundTasks):
    user = create_user(payload)
    bg.add_task(send_confirmation_email, user.email)
    return user

from fastapi import WebSocket

Case Study: Employee CRUD API

This repository is one practical CRUD example. Use it as a case study after learning the general FastAPI patterns from the earlier sections.

Theory deep dive

A CRUD API models the lifecycle of a resource. The resource here is an employee. Create adds a new employee, Read fetches employees, Update changes an existing employee, and Delete removes one. Most business APIs are built from this same pattern.

The current implementation is intentionally simple: it uses an in-memory list so you can focus on routing, validation, and errors first. Once those concepts are clear, the same API contract can be backed by a real database.

Application file

main.py creates the FastAPI app, registers routes, stores employees in memory, and raises HTTP errors.

Schema file

model.py defines Pydantic models for create, update, and response validation.

Storage style

employees_db is a Python list. It is simple for learning, but data disappears when the server restarts.

Database Upgrade: From List To SQLite

Learning APIs often start with an in-memory list. That is useful for route practice, but real APIs need persistence, constraints, relationships, transactions, and query performance. SQLite is the easiest next step before PostgreSQL or MySQL.

Theory deep dive

A database is responsible for durable storage, constraints, indexing, and safe querying. In-memory lists are process-local and temporary; databases are designed to keep data consistent across requests and restarts.

When you add SQLAlchemy, remember the separation of responsibilities: SQLAlchemy models describe tables, Pydantic models describe API data, and CRUD functions translate between HTTP requests and database operations.

Why replace the list?

  • Data survives server restarts.
  • IDs, emails, usernames, slugs, or SKUs can be unique at the database level.
  • Queries can filter, sort, and paginate efficiently.
  • The API becomes closer to real production design.

Install database dependencies

pip install sqlalchemy

Database model

from sqlalchemy import create_engine, Column, Integer, String
from sqlalchemy.orm import DeclarativeBase, sessionmaker

DATABASE_URL = "sqlite:///./app.db"
engine = create_engine(DATABASE_URL, connect_args={"check_same_thread": False})
SessionLocal = sessionmaker(bind=engine, autoflush=False, autocommit=False)

class Base(DeclarativeBase):
    pass

class Product(Base):
    __tablename__ = "products"

    id = Column(Integer, primary_key=True, index=True)
    name = Column(String(80), nullable=False)
    sku = Column(String(40), unique=True, nullable=False, index=True)
    category = Column(String(40), nullable=False)

Base.metadata.create_all(bind=engine)

Database dependency

from typing import Generator

def get_db() -> Generator:
    db = SessionLocal()
    try:
        yield db
    finally:
        db.close()

Create with database session

from fastapi import Depends, HTTPException
from sqlalchemy.orm import Session

@app.post("/products", response_model=ProductResponse, status_code=201)
def create_product(payload: ProductCreate, db: Session = Depends(get_db)):
    existing = db.get(Product, payload.id)
    if existing:
        raise HTTPException(status_code=400, detail="Product ID already exists")

    product = Product(**payload.model_dump())
    db.add(product)
    db.commit()
    db.refresh(product)
    return product

Read with pagination

@app.get("/products", response_model=list[ProductResponse])
def list_products(
    offset: int = 0,
    limit: int = 20,
    db: Session = Depends(get_db),
):
    return db.query(Product).offset(offset).limit(limit).all()

Important: API schemas and database models are different responsibilities. Pydantic schemas validate external data; SQLAlchemy models represent database tables.

Authentication And Authorization

Authentication answers "Who are you?" Authorization answers "What are you allowed to do?" FastAPI supports standard security patterns through dependencies.

Theory deep dive

Authentication protects identity. Authorization protects actions. A user may be signed in but still not allowed to delete records, approve payments, view private data, or access admin screens.

JWT-based auth works well for APIs because the client can send a bearer token with every request. The server verifies the token, extracts the user identity and role, and lets dependencies decide whether the endpoint should run.

Typical JWT flow

1
User sends username and password to /token.
2
Server verifies password hash.
3
Server returns a signed JWT access token.
4
Client sends Authorization: Bearer <token> on protected requests.
5
FastAPI dependency validates the token before running the endpoint.

Install auth dependencies

pip install python-jose[cryptography] passlib[bcrypt] python-multipart

Password hashing

from passlib.context import CryptContext

pwd_context = CryptContext(schemes=["bcrypt"], deprecated="auto")

def hash_password(password: str) -> str:
    return pwd_context.hash(password)

def verify_password(plain_password: str, hashed_password: str) -> bool:
    return pwd_context.verify(plain_password, hashed_password)

Testing FastAPI Properly

Testing proves that your API contract works. A complete test suite checks successful requests, validation errors, business-rule errors, authentication errors, and edge cases.

Theory deep dive

API tests should verify behavior, not implementation details. A test should send a request, inspect the status code, and check the JSON response. This gives confidence that clients can use the API correctly.

Good test coverage includes the happy path and failure paths. Duplicate IDs, invalid enum values, invalid emails, missing records, update errors, delete errors, and auth failures are just as important as successful creation.

Production Readiness

Production is not just "run the server." A production API needs configuration, logging, persistence, security, tests, and a repeatable deployment process.

Theory deep dive

Production readiness means the API can run reliably outside your laptop. It should recover from restarts, keep secrets out of code, log useful information, reject unsafe requests, and be deployed the same way every time.

The usual production priorities are persistent storage, authentication, environment configuration, automated tests, structured logging, observability, and a deployment target such as Docker, Render, Railway, AWS, or another platform.

Interview And Revision Notes

Use this section for structured revision from basic to advanced. A strong answer should define the concept, explain why it matters, give a practical example, and mention a production consideration.

Answer framework

For interviews, do not only give definitions. Connect the concept to API behavior: request parsing, validation, business logic, response serialization, error handling, persistence, testing, and deployment.

A good answer format is: definition -> reason -> example -> production note. Example: Pydantic validates incoming JSON, prevents bad data from entering business logic, can calculate BMI using computed fields, and should be paired with response models to avoid leaking private fields.

Basic level

QuestionStrong answer
What is FastAPI?A Python ASGI framework for building APIs. It uses Starlette for web handling and Pydantic for validation, serialization, and OpenAPI schema generation.
Why is FastAPI popular?It is fast, type-hint driven, easy to write, supports async, validates data automatically, and generates Swagger/ReDoc documentation without extra setup.
What is an endpoint?An endpoint is a URL plus HTTP method mapped to a function, such as GET /products/{product_id}.
What are path parameters?Values embedded in the URL path, usually used to identify a resource, such as product_id in /products/{product_id}.
What are query parameters?Values after ? in the URL, usually used for filtering, sorting, searching, and pagination, such as ?limit=20.
What is a request body?Structured data sent by the client, usually JSON, commonly used with POST, PUT, and PATCH.
What is Pydantic?A data validation and serialization library. FastAPI uses it to validate request bodies and shape responses.
What is Uvicorn?An ASGI server that runs FastAPI applications. In development, you commonly run uvicorn main:app --reload.

Intermediate level

QuestionStrong answer
What is response_model?It validates and serializes outgoing responses, documents the response schema, and prevents accidental extra fields from being returned.
Why does FastAPI return 422?The request was syntactically understood, but validation failed. Examples include invalid email, wrong type, missing required field, or invalid enum value.
What is dependency injection?A way to declare reusable dependencies such as DB sessions, auth, settings, or permissions using Depends(). FastAPI resolves them before calling the endpoint.
What is the difference between PUT and PATCH?PUT usually replaces a full resource, while PATCH partially updates only provided fields.
How do you raise an API error?Use HTTPException with a status code and detail, for example raise HTTPException(status_code=404, detail="Not found").
How do you handle CORS?Add CORSMiddleware and configure trusted origins, methods, headers, and credentials. Avoid wildcard origins in sensitive production APIs.
How do you test FastAPI routes?Use TestClient(app), send requests to endpoints, and assert status codes and response bodies.
How can Pydantic calculate BMI?Accept height_cm and weight_kg, then use @computed_field to return derived fields like bmi and bmi_category.

Advanced level

QuestionStrong answer
How does FastAPI handle async?FastAPI supports async def endpoints for non-blocking I/O. Use async database drivers or HTTP clients inside async routes; avoid blocking calls in the event loop.
When should you use sync vs async endpoints?Use sync def for CPU-bound or blocking library calls. Use async def for async I/O such as async DB calls, HTTP calls, WebSockets, and streaming.
What happens in a dependency with yield?Code before yield runs before the endpoint. Code after yield runs as cleanup, useful for closing DB sessions or releasing resources.
How do you structure a large FastAPI project?Split routers, schemas, models, CRUD/service logic, settings, security, database setup, and tests into separate modules.
How do you prevent leaking sensitive fields?Use separate request and response schemas. Never return ORM models directly when they contain password hashes, tokens, or internal fields.
How do you secure protected routes?Use OAuth2/JWT dependencies to identify the user, then role or permission dependencies to authorize actions.
How do you make FastAPI production-ready?Add persistent storage, migrations, auth, environment config, logging, tests, CORS restrictions, monitoring, deployment automation, and secret management.
How would you debug slow endpoints?Add timing middleware, inspect database queries, check blocking calls inside async routes, profile slow code, and monitor latency metrics.

Scenario questions

ScenarioHow to answer
Client sends age, height, and weight. API should return BMI.Create a Pydantic model with validated numeric fields and computed fields for bmi and category. Add a POST /health/bmi route.
Frontend cannot call the API from browser.Check CORS. Configure CORSMiddleware with the frontend origin and required methods/headers.
Data disappears after server restart.The app is using in-memory storage. Add a database such as SQLite/PostgreSQL and move CRUD logic to database-backed functions.
Endpoint returns password hash in response.Create a separate response schema that excludes sensitive fields and use it as response_model.
Duplicate username should fail.Check uniqueness in business logic and enforce a unique database constraint. Return 400 or 409 with a useful message.

Common mistakes

  • Using GET for actions that create, update, or delete data.
  • Putting database models directly in public responses without response schemas.
  • Storing secrets in source code.
  • Using in-memory lists and expecting data to persist after restart.
  • Allowing allow_origins=["*"] in production without understanding the risk.
  • Writing only happy-path tests and skipping validation/error tests.
  • Using blocking database or HTTP calls inside async def endpoints.
